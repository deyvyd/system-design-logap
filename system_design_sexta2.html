Elasticidade
03 / 10

Auto Scaling & Elasticidade

Scale out com Load Balancer resolve o dia a dia. Mas quem decide quando subir ou descer instâncias?

📈

Trigger de scale-out

Métricas: CPU > 70% por 2 min, latência > 500ms, fila > 1000 mensagens. O Auto Scaling Group sobe novas instâncias automaticamente.

⚙️

Min / Max / Desired

Configuração: mínimo 2 instâncias (redundância), máximo 10 (custo), desejado parte de 2. A Black Friday pode atingir o máximo em minutos.

📉

Scale-in (cooldown)

Quando o tráfego cai, instâncias sobressalentes são terminadas. Cooldown evita oscilação: aguarda X segundos antes de decidir.

Stateless é requisito: se a instância guarda sessão de usuário em memória, scale-out quebra. Estado deve estar no banco ou no Redis.

Alta Disponibilidade
04 / 10

Multi Data Center

Um único data center — mesmo com auto scaling — ainda é um ponto único de falha geográfico.

🔵

Active / Active

Ambos os DCs atendem tráfego simultaneamente. O Load Balancer global (GeoDNS) roteia para o mais próximo. Se um cai, o outro absorve tudo.

🟡

Active / Passive (DR)

O DC secundário (standby) só assume em caso de falha do primário. Mais simples, mas há latência no failover manual ou automatizado.

⚡

Replicação de dados

Banco precisa replicar entre regiões. Consistência eventual é o trade-off aceitável — eventual consistency cross-region é a norma.

RTO vs RPO: Recovery Time Objective (tempo até voltar) e Recovery Point Objective (quanto dado posso perder). Multi DC reduz ambos drasticamente.

Mensageria
05 / 10

Mensageria Assíncrona

Quando um serviço chama outro diretamente (síncrono), uma falha propaga para toda a cadeia. A fila desacopla isso.

🔴

Síncrono (REST/gRPC)

API chama outro serviço e aguarda resposta. Se o downstream estiver lento ou fora, o upstream trava. Acoplamento forte.

🟢

Assíncrono (Fila/Tópico)

Produtor publica na fila e segue. Consumer processa no próprio ritmo. Falha no consumer não derruba o produtor.

🛠️

Ferramentas

RabbitMQ: filas simples, roteamento flexível. Kafka: alto volume, replay de eventos, log imutável. SQS: managed na AWS.

Kafka > fila simples? Só quando você precisa de replay, múltiplos consumers independentes, ou volume > 1M mensagens/s. Do contrário, RabbitMQ/SQS é mais simples e suficiente.

Teorema CAP
06 / 10

CAP & PACELC

Em sistemas distribuídos, você só consegue garantir 2 das 3 propriedades do Teorema CAP.

CP — Consistency + Partition Tolerance

Prioriza dados corretos. Em partição, sistema pode recusar leituras/escritas para não servir dado desatualizado. PostgreSQL, Redis, HBase.

AP — Availability + Partition Tolerance

Prioriza estar disponível. Em partição, continua respondendo mesmo com dado potencialmente desatualizado. Cassandra, DynamoDB, CouchDB.

PACELC — extensão do CAP: quando não há partição (Else), o trade-off passa a ser entre Latência e Consistência. Cassandra: configura nível de consistência por query.
Na prática: Dados financeiros → CP (PostgreSQL). Feed de atividades → AP (Cassandra). Cache → CP (Redis). Configurações globais → AP com TTL curto.

Microsserviços
07 / 10

Microsserviços — a armadilha

Microservices viraram sinônimo de "arquitetura moderna". Mas a Amazon Prime voltou para monolito e ficou 90% mais barato.

MONOLITO Tudo num processo só. Deploy único. Mais simples de debugar, testar, versionar. Funciona muito bem até escala alta.
MODULAR Monolito com módulos bem definidos. Separação lógica, deploy ainda unificado. Sweet spot para a maioria dos casos.
MICROSERV. Cada serviço independente. Deploy separado. Escala independente. Mas: rede entre serviços, rastreamento distribuído, complexidade operacional alta.
Amazon Prime Video (2023): migrou de microsserviços para monolito modular. Custo operacional caiu 90%. Throughput aumentou. Complexidade de rede era o gargalo.
Quando microsserviços fazem sentido: times grandes e autônomos (Netflix, Uber), serviços com requisitos radicalmente diferentes de escala, linguagens distintas por domínio.

08 / 10

Sharding

Read replicas resolvem gargalo de leitura. Mas e quando a tabela tem 1 bilhão de registros? Sharding particiona os dados.

🔪

O que é sharding?

Dividir os dados horizontalmente entre múltiplos bancos (shards). Cada shard tem um subconjunto das linhas. Escrita e leitura distribuídas.

🔑

Shard key — decisão crítica

Define como distribuir dados. Ruim: campo de baixa cardinalidade (país) → hotspot. Bom: user_id (hash) → distribuição uniforme.

🌡️

Problema do hotspot

Se 80% das requisições vão para o shard A, perdemos os benefícios. Range sharding (A-M / N-Z) é vulnerável a hotspot. Hash sharding distribui melhor.

Sharding é complexo: joins cross-shard viram chamadas de rede, transações distribuídas são difíceis, rebalanceamento é custoso. Esgote outras opções antes: índices, cache, read replicas, particionamento de tabela.

Case Real
09 / 10
Projeto pessoal · em desenvolvimento

MoneyTalks

App de finanças pessoais. React 19 · TypeScript · Supabase · PostgreSQL com 14 tabelas. Onde cada decisão de system design foi consciente.

🎯

PostgreSQL (CP) — por quê?

Dados financeiros exigem consistência forte. Saldo não pode ser lido desatualizado. ACID é requisito, não opcional. Cassandra seria a escolha errada.

⚡

Supabase Edge Functions

Processamento de regras de negócio próximo ao banco. Reduz round-trips entre cliente e DB. RLS (Row Level Security) garante que usuário só acessa seus dados.

🧩

14 tabelas — decisões conscientes

Sem JSONB para campos semi-estruturados (auditoria usaria), normalização completa para evitar inconsistência, índices nos campos de filtro mais comuns.

O que não foi feito (ainda): sharding (volume não justifica), cache (Supabase serve via edge), mensageria (ainda síncrono). System design é sobre quando adicionar complexidade, não poder adicionar.

Separação de responsabilidades, inversão de dependência, monolito modular bem estruturado. Fundação antes de partir para microserviços.
