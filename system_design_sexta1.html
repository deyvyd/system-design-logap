O que é System Design

É o processo de tomar decisões de arquitetura antes de escrever código — entendendo o problema para criar a solução adequada.

Overkill é tão ruim quanto subdimensionado. Um sistema com microsserviços para 50 usuários é desperdício de recursos e complexidade desnecessária.
🎯

Fit, não template

Não existe arquitetura certa em abstrato. Existe a arquitetura certa para aquele problema, naquele volume, naquele momento.

⚖️

Trade-offs, não balas de prata

Toda decisão arquitetural resolve um problema e cria outro. O objetivo é escolher conscientemente qual problema você prefere ter.

🔄

Evolução gradual

Você não constrói a arquitetura final de uma vez. Ela evolui conforme o sistema cresce e os gargalos aparecem.

05 / 10

RF e RNF como ponto de partida

Toda arquitetura foi feita para atender requisitos. Sem eles, não existe system design — existe chute.

⚙️ Requisitos Funcionais

  • → O que o sistema faz
  • → Usuário pode fazer upload
  • → Sistema gera relatório
  • → Notificação por e-mail

📐 Requisitos Não-Funcionais

  • → Como o sistema se comporta
  • → Suporte 10k usuários simultâneos
  • → Resposta em até 200ms
  • → 99,9% de disponibilidade
RF define as funcionalidades e serviços. "Usuário faz upload" → precisamos entender o que o usuário faz no sistema para saber quais funcionalidades construir e serviços oferecer.
RNF define como o sistema se comporta. "Leitura em até 200ms" não é uma feature, é uma exigência de qualidade sobre algo que já existe. CDN + Cache na frente do storage. O arquivo sai do servidor de borda mais próximo do usuário, não do S3 direto.

Vertical vs Horizontal

Dois caminhos para crescer — com trade-offs muito diferentes.

⬆️ Vertical (Scale Up)

  • + Simples de implementar
  • + Sem mudança na aplicação
  • − Tem limite físico de hardware
  • − Não resolve disponibilidade
  • − Ponto único de falha

↔️ Horizontal (Scale Out)

  • + Sem limite teórico
  • + Redundância e failover
  • + Se um cai, outros assumem
  • − Precisa de Load Balancer
  • − Banco vira o gargalo
Load Balancer distribui as requisições entre instâncias, fica exposto publicamente enquanto os servidores ficam em rede privada. Faz health check: se uma instância cai, para de enviar tráfego para ela automaticamente.

Replicação de Banco

Com múltiplas instâncias da aplicação, o banco vira o gargalo. A solução é separar escrita de leitura.

✍️

Master (escrita)

Recebe todos os INSERTs, UPDATEs e DELETEs. É a fonte de verdade dos dados.

📖

Slaves (leitura)

Réplicas do master. Recebem os SELECTs. A maioria das aplicações lê muito mais do que escreve.

⏱️

Consistência eventual

A replicação leva alguns milissegundos. Um dado recém-inserido pode não aparecer imediatamente nos slaves. Isso é um trade-off consciente.

Quando isso é problema? Saldo bancário não pode estar desatualizado. Contador de likes em rede social pode. O contexto define a tolerância.

08 / 10

Cache-Aside Pattern

Quando o banco é bombardeado com consultas repetidas dos mesmos dados, o cache resolve sem mudar a infraestrutura.

01 Requisição chega na API
02 Consulta o Redis primeiro. O dado existe?
03a Cache hit → retorna direto. Banco não é tocado.
03b Cache miss → vai ao banco, armazena no Redis com TTL, retorna.
04 Quando TTL expira, próxima requisição atualiza o cache.

Cachear apenas dados que são:

🔁Muito requisitados (consultas repetidas)
🕰️Não mudam com frequência alta
📊Tolerantes a leve desatualização

Case Real
09 / 10
Projeto interno · Aiyra

DocGen

Automação na geração de documentações técnicas e de desenvolvimento para o grupo Axia. Redução significativa no tempo de geração para quem adotou a ferramenta.

Decisões de arquitetura tomadas:

JSON em vez de banco de dados — dados são temporários (só para geração do documento). Banco seria complexidade desnecessária. Isso é system design: escolher a ferramenta certa para o problema certo.
JSON reutilizável — doc incompleta pode ser retomada. Estrutura pode ser reaproveitada para novo documento. Dado o problema, essa foi a solução de persistência adequada.
